# Гайдлайн розмітки (v2: tiers + pixel thresholds + обов'язкові метадані)

> Формат: YOLO bbox (`class_id x_center y_center width height`, нормалізовано 0-1).
> Тулза: CVAT (self-hosted) або Roboflow.
> OBB можливо на v2 — закладаємо опційно, див. розділ нижче.

## Pixel thresholds (НОВЕ — критично)

Базуємось на GSD (`docs/pixel_budget.md`):

| Категорія | Мінімум по меншій стороні bbox | Дія |
|---|---:|---|
| Можна детектувати | ≥10–12 px | мітимо |
| Можна класифікувати | ≥20 px | впевнено мітимо |
| Нижче порога | <10 px | **НЕ мітимо** (це шум, не сигнал) |

Якщо об'єкт на кадрі менший — пропускаємо. Це залежить від реальної камери (буде уточнено після фіксації параметрів).

## Метадані кадру — ОБОВ'ЯЗКОВО (НОВЕ)

Кожен кадр **мусить** мати JSON sidecar `<image_stem>.json`. Без цього кадр не приймається в датасет.

```json
{
  "image_id": "img_00123",
  "source": "synthetic|real",
  "altitude_m": 400,
  "view_angle_deg": 25,
  "tier_mode": "tier1_only|tier1_tier2",
  "season": "summer|winter|mud_season",
  "landscape": "forest_belt|forest|field|mud|urban|road",
  "weather": "clear|fog|rain|snow",
  "time_of_day": "day|dusk|night",
  "camera": {
    "sensor_res": [1920, 1080],
    "hfov_deg": 70,
    "modality": "EO|IR"
  }
}
```

**Чому обов'язково:**
- Без `altitude_m` не зможемо стратифікувати спліт за висотою. Модель навчиться на 300 м і осліпне на 800.
- Без `view_angle_deg` не зрозуміємо чи oblique vs nadir впливає на per-class mAP.
- Без `source` не зможемо окремо оцінювати real vs synthetic.

Для **синтетики:** генератор знає висоту, кут, FOV, GSD — мусить писати їх автоматично. Це майже безкоштовно.
Для **реальних даних:** дорого, але необхідно. Якщо нема telemetry — оцінка людиною (300 ±50, 400 ±100, 800 ±200).

## Bounding box

- **Tight fit:** торкатись усіх країв, без зайвого «повітря». Допустимий запас ≤2 px.
- **Цілий силует:** включити башточку/стволи/антени.
- **Тіні і пил:** НЕ включати.
- **Pixel threshold:** див. таблицю вище — нижче ~10 px НЕ мітимо.

## Truncation (край кадру)

- ≥25% видимо — мітимо по **amodal** межах (повний гіпотетичний силует, не лише видима частина).
- <25% — пропускаємо.

## Occlusion (перекриття деревами/кущами/сітками/технікою)

Best practice aerial-detection (VOD-UAV occlusion levels; occlusion-review arXiv 2101.08845):

- **≥25% видимо** — мітимо за **amodal** межами (bbox по повному силуету, ніби перекриття немає).
  Це критично: modal-bbox (лише видимі пікселі) «стрибає» від кадру до кадру і псує IoU/навчання.
- **<25% видимо** — пропускаємо (шум, не сигнал).
- Поріг **25%** (не 50%): реальна техніка часто перекрита посадкою на 40-70%, і модель мусить її знаходити.

**Синтетика рахує це автоматично.** `render_runner` two-pass: рендер БЕЗ оклюдерів (повний силует
`vehicle_full_px`) + рендер З оклюдерами (`vehicle_visible_px`); `visibility_fraction` пишеться у
metadata sidecar, bbox береться з amodal-силуету, кадри з visibility < `occlusion.min_visible_frac`
(0.25) відкидаються. Оклюдери — дерева/кущі/маскувальні сітки на лінії зору камера→ціль
(`occlusion:` блок конфіга).

## Tier-залежні правила

- **Tier 1 кадри** (altitude ≥400 м, особливо 800 м): мітимо тільки `tank, ifv_apc, artillery, air_defense, mlrs, truck_logistics, radar_ew, destroyed_vehicle`. Tier 2 ігноруємо (вони <10 px у режимі A).
- **Tier 2 кадри** (altitude ≤300 м або режим zoom): мітимо всі 11 класів.
- Якщо аннотатор бачить `infantry` на 800 м кадрі — **не мітити**, це <2 px = шум.

## Кілька об'єктів

- Колона техніки в русі — кожна машина окремий бокс.
- `infantry` група — один бокс на компактну групу. Якщо група розтягнута на >2× ширини найбільшої машини — кілька боксів.

## Спалена техніка (ЗМІНА проти v1)

Раніше була hard negative. Зараз — **окремий клас** `destroyed_vehicle`. Мітимо її.
Критерій «спалена» (BDA-ознаки зверху — BDA Wikipedia; Oryx methodology):
- **turret toss** — башта зірвана/поруч/перевернута (детонація БК; ~½ втрат — catastrophic kill);
- вигорілий/чорний корпус, копоть, дим, вирва поруч;
- відсутні гусениці/колеса/башта → **розірваний силует** (головна відмінність від цілої зверху).

Якщо в кадрі є і спалена, і боєздатна — мітимо обидві (різними класами).

**Синтетика:** `destroyed:` блок конфіга генерує wreck процедурно (перекид корпусу +
обгорілий матеріал). `mode: hn` — рендерить брухт БЕЗ боксу (anti-false-positive), `mode: class` —
мітить своїм `class_id` (детектований клас `destroyed_vehicle`).

## Танк vs БТР/ББМ зверху (ознаки для анотатора)

При достатньому масштабі (≥20px — поріг класифікації):
- **Танк** — довгий **ствол, що виступає за корпус** (найнадійніша ознака зверху), широка башта, **гусениці**.
- **БТР** — немає виступаючого ствола (кулемет/модуль на відкритій платформі), **колеса 8×8**, човноподібний корпус.
- На <20px ствол = 1-2px і не розрізняється → на крейсерських висотах танк/БТР зливаються.
  Тут допомагає **OBB** (`output.bbox_format: obb`): орієнтований bbox кодує видовженість/ствол.

## Сезонність і ландшафти

Збір повинен покривати:
- **Сезони:** літо, зима (сніг), міжсезоння (багнюка).
- **Ландшафти:** лісосмуга, ліс, поле, ґрунтова дорога з багнюкою, місто, асфальтована дорога.
- **Висоти:** 300, 400, 800 м (мінімум 3 точки в кожному спліті).
- **Кути:** діапазон 10–60° від горизонту.

## Стратифікація сплітів (НОВЕ — критично)

Train/val/test мусять бути стратифіковані не тільки за класом, а **за класом × висотою × кутом**. Інакше:
- Висота: модель навчиться на 300 м і провалить 800 м.
- Кут: модель навчиться на «прямому профілі» і провалить «зверху».

Скрипт `dataset/split.py` (буде) робить stratified split з seed.

## OBB (опційно, v2)

Для oblique view + витягнутої техніки axis-aligned bbox тягне багато фону. На дальніх дрібних цілях це псує localization (IoU падає бо bbox величезний відносно реального об'єкта).

Опція на v2: переходимо на OBB (4 точки замість 2 кутів). YOLO11-obb це підтримує.

На v1 використовуємо звичайні axis-aligned bbox для швидкості розмітки і простоти. Закладаємо в анотаційний tool можливість експорту в обидва формати.

## Контроль якості

- **Double-pass:** перші 100 кадрів кожного нового аннотатора рев'юить senior.
- **IoU consistency check:** на 5% кадрів повторна розмітка іншим аннотатором, цільовий IoU ≥0.85.
- **Pixel size sanity check:** автоматичний скрипт `dataset/qa_pixel_size.py` (буде) — flag bboxes < 10 px у `qa_log.md`.
- Помилкові розмітки логуються в `dataset/qa_log.md`.

## Заборонене

- Не мітити об'єкти <10 px (це шум).
- Не міняти ID класу заднім числом без міграції.
- Не зменшувати box щоб не залазив на сусідній — boxes можуть перекриватись.
- Не вгадувати клас якщо не впевнений: тег `unsure` + рев'ю.
- Не приймати кадр без metadata sidecar.
