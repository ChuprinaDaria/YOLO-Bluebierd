# Гайдлайн розмітки (v2: tiers + pixel thresholds + обов'язкові метадані)

> Формат: YOLO bbox (`class_id x_center y_center width height`, нормалізовано 0-1).
> Тулза: CVAT (self-hosted) або Roboflow.
> OBB можливо на v2 — закладаємо опційно, див. розділ нижче.

## Контекст продукту — навіщо ці пороги (2026-06-24)

Зафіксовано у командному чаті (Гіряков Володимир, CV Engineer; Гаврилюк Дмитро, Head of R&D Software). Пороги нижче — не довільні, вони задають межу між **нашою територією** (YOLO autonomous target selection) і **територією конкурентів** (pixel-lock SOT).

**Конкурент ("АВТО" у маркетингу) = SOT на 5-10 px:**
- Пілот вручну захоплює ціль у 5-10 px, борт летить туди, якщо ціль статична.
- На динамічних цілях SOT зривається 100%. "Чистий маркетинг — не виживе на реальному ринку" (Володимир).

**Наш справжній АВТО = YOLO + autonomous target selection by class priority:**
- Борт сам обирає ціль за пріоритетом класів і наводиться без пілота.
- Це вимагає **bbox ≥20-30 px** (нижче — SOT-зона, ми туди не лізем).
- Дмитро: треба object segmentation + ігнорування деформації через кут/рух, не pixel-lock.

**Операційний сценарій (де АВТО вмикається):**
- Борт чисто наводиться з **~300 м** висоти (відео+керування ок).
- На **~100 м** відеоканал деградує ("сніг на екрані", керування лагає) — це "за радіогоризонтом" у формулюванні Дмитра (не повна втрата керування, а саме деградація відео через кривину Землі / дерева / рельєф).
- Останні метри борт мусить довести **автономно** — це і є терминальна стадія, наш core use case.

**Наслідок для розмітки:** мінімум 20 px — non-negotiable. Нижче — це шум для нашої моделі (борт не зможе автономно навестись), і конкурентська територія, де ми гірші за заточений SOT.

## Pixel thresholds (критично)

Базуємось на GSD (`docs/pixel_budget.md`) + business context вище:

| Категорія | Мінімум по меншій стороні bbox | Дія | Чому |
|---|---:|---|---|
| Можна класифікувати | ≥20 px | **впевнено мітимо** | наша робоча зона — autonomous selection працює |
| Borderline | 10-19 px | мітимо, але tag `borderline` | може потрапити у train, але приоритет нижчий |
| Нижче порога | <10 px | **НЕ мітимо** | SOT-територія, не наша; шум для YOLO |

Якщо об'єкт на кадрі менший — пропускаємо. Це залежить від реальної камери (буде уточнено після фіксації параметрів).

## Метадані кадру — ОБОВ'ЯЗКОВО (НОВЕ)

Кожен кадр **мусить** мати JSON sidecar `<image_stem>.json`. Без цього кадр не приймається в датасет.

```json
{
  "image_id": "light_vehicle_00007",
  "source": "synthetic_blender|real",
  "dataset_version": "df-v1.0.0",
  "distance_m": 400,
  "view_angle_deg": 25,
  "altitude_m": 169,
  "horizontal_distance_m": 363,
  "hfov_deg": 15,
  "sensor_res": [640, 640],
  "modality": "EO",
  "tier_mode": "tier1_only|tier1_tier2",
  "season": "summer|autumn_mud|winter|spring",
  "landscape": "forest_belt|forest|field|dirt_road|urban|road",
  "weather": "clear|fog|haze|rain|snow",
  "time_of_day": "day|dusk|night",
  "class_name": "light_vehicle",
  "class_id": 7,
  "n_boxes": 1,
  "has_targets": true
}
```

**Primary параметр `distance_m`** — пряма 3D line-of-sight камера→техніка (200-1000 м для типового drone). `altitude_m` і `horizontal_distance_m` — derived з `distance_m × sin(view_angle)` / `× cos(view_angle)`, пишуться для зручності спліту/аналізу.

**Чому обов'язково:**
- Без `distance_m` не зможемо стратифікувати спліт за дальністю. Модель навчиться на 300 м і осліпне на 800.
- Без `view_angle_deg` не зрозуміємо чи oblique vs nadir впливає на per-class mAP.
- Без `source` не зможемо окремо оцінювати real vs synthetic.
- Без `hfov_deg` не реконструюємо px-розмір при зміні sensor configs.

Для **синтетики:** генератор знає distance, кут, FOV, GSD — мусить писати їх автоматично (вже реалізовано у `render_runner.py`).
Для **реальних даних:** дорого, але необхідно. Якщо нема telemetry — оцінка людиною (300 ±50, 400 ±100, 800 ±200).

## Bounding box

- **Tight fit:** торкатись усіх країв, без зайвого «повітря». Допустимий запас ≤2 px.
- **Цілий силует:** включити башточку/стволи/антени.
- **Тіні і пил:** НЕ включати.
- **Pixel threshold:** див. таблицю вище — нижче ~10 px НЕ мітимо.

## Truncation

- ≥50% видимо — мітимо по видимій частині.
- <50% — пропускаємо.

## Occlusion

- ≤50% перекрито — мітимо за гіпотетичними межами.
- >50% — пропускаємо.

## Tier-залежні правила

- **Tier 1 кадри** (`distance_m` ≥400 з hfov 15°): мітимо тільки `tank, ifv_apc, artillery, air_defense, mlrs, truck_logistics, radar_ew`. Tier 2 ігноруємо (вони <10 px при цьому px-budget). Спалена техніка — без боксу як hard negative.
- **Tier 2 кадри** (`distance_m` ≤300 з hfov 15° або zoom-mode hfov ≤30°): мітимо всі 10 класів.
- Якщо аннотатор бачить `infantry` на distance 800 м кадрі — **не мітити**, це <2 px = шум.

## Кілька об'єктів

- Колона техніки в русі — кожна машина окремий бокс.
- `infantry` група — один бокс на компактну групу. Якщо група розтягнута на >2× ширини найбільшої машини — кілька боксів.

## Спалена техніка

**Hard negative, БЕЗ боксу** (per `docs/taxonomy_v3.md` §"Hard negatives"). Рішення зафіксоване 2026-06-24 у плані abundant-snacking-thacker.

Раніше була `destroyed_vehicle` як окремий клас (v2). Відкинуто: 11 класів роздуває label space на marginal value (operators цілять у боєздатне, не у вигоріле), і borderline-кадри (живий vs wreck) даватимуть систематичну плутанину bbox.

Критерій «спалена» (для генерації hard negative кадрів без боксу): вигоріла, відсутні гусениці/колеса/башта, дим/копоть.

Якщо в кадрі є і спалена, і боєздатна — мітимо **тільки боєздатну**, спалена існує у фоні без боксу.

## Сезонність і ландшафти

Збір повинен покривати:
- **Сезони:** літо, осінь (багнюка), зима (сніг), весна.
- **Ландшафти:** лісосмуга, ліс, поле, ґрунтова дорога з багнюкою, місто, асфальтована дорога.
- **Distance:** 150-1000 м (стратифіковано: дрібні класи зрізають на 500м, великі тримають до 1000м — див. `docs/pixel_budget.md`).
- **Кути:** 10-30° oblique + 90° nadir для diversity ракурсів — не для видимості.

Геометрія обумовлена операційним сценарієм (див. "Контекст продукту"): модель тренується на тому, що борт бачить перед автономним доведенням з ~300 м до ~100 м.

## Стратифікація сплітів (НОВЕ — критично)

Train/val/test мусять бути стратифіковані не тільки за класом, а **за класом × distance × кутом × сезоном**. Інакше:
- Distance: модель навчиться на 300 м і провалить 800 м.
- Кут: модель навчиться на «прямому профілі» і провалить «зверху».
- Сезон: модель навчиться на літо і провалить зиму.

Скрипт `dataset/split.py` (буде) робить stratified split з seed.

## OBB (опційно, v2)

Для oblique view + витягнутої техніки axis-aligned bbox тягне багато фону. На дальніх дрібних цілях це псує localization (IoU падає бо bbox величезний відносно реального об'єкта).

Опція на v2: переходимо на OBB (4 точки замість 2 кутів). YOLO11-obb це підтримує.

На v1 використовуємо звичайні axis-aligned bbox для швидкості розмітки і простоти. Закладаємо в анотаційний tool можливість експорту в обидва формати.

## Контроль якості

- **Double-pass:** перші 100 кадрів кожного нового аннотатора рев'юить senior.
- **IoU consistency check:** на 5% кадрів повторна розмітка іншим аннотатором, цільовий IoU ≥0.85.
- **Pixel size sanity check:** автоматичний скрипт `dataset/qa_pixel_size.py` (буде) — flag bboxes < 10 px у `qa_log.md`.
- Помилкові розмітки логуються в `dataset/qa_log.md`.

## Заборонене

- Не мітити об'єкти <10 px (це шум).
- Не міняти ID класу заднім числом без міграції.
- Не зменшувати box щоб не залазив на сусідній — boxes можуть перекриватись.
- Не вгадувати клас якщо не впевнений: тег `unsure` + рев'ю.
- Не приймати кадр без metadata sidecar.
